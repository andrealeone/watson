# Watson CLI Demos

A collection of example CLIs built with Watson, each demonstrating a different
feature or pattern. They are validated end-to-end by `tests/demos/demos.test.ts`.

## The real API at a glance

A Watson command is a **`CommandModule`** — a plain object with optional `meta`,
`flags`, `args`, and a `run(ctx)` function. Commands are mapped to routes in a
**manifest**, and the **`run()`** dispatcher resolves argv → command, parses flags,
builds the `Context`, and calls `run`. Return a number from `run` to set the exit code.

```typescript
import type { CommandModule } from '../../../src/types/command'
import type { Config } from '../../../src/types/config'
import { defineManifest, run } from '../../../src/core/runtime'

const hello: CommandModule = {
  meta: { description: 'Greet someone' },
  run(ctx) {
    ctx.io.write(`Hello, ${ctx.positionals[0] ?? 'World'}!`)
  },
}

const manifest = defineManifest({ hello })
const config: Config = { name: 'demo', bin: 'demo', commandsDir: 'commands', version: '1.0.0' }
run(manifest, config).then((code) => process.exit(code))
```

The `Context` passed to every command exposes:

| Field             | Description                                              |
| ----------------- | -------------------------------------------------------- |
| `ctx.positionals` | non-flag arguments after the route                       |
| `ctx.flags`       | parsed + coerced flags (per the command's `flags` spec)  |
| `ctx.env`         | environment variables                                    |
| `ctx.cwd`         | working directory                                        |
| `ctx.io`          | `write` / `writeError` / `colour` / `spinner` / `prompt` |
| `ctx.logger`      | `debug` / `info` / `warn` / `error`                      |

> Routes are slash-delimited in `defineManifest` (`'users/list'`), and the router
> matches the **longest** prefix — so `users get 1` resolves to the `users/get`
> command with `['1']` left as positionals.

> In a real project the manifest is code-generated by scanning a commands directory.
> Smaller demos use `defineManifest` for simplicity; larger demos use `discoverManifest`
> to automatically load commands from separate files.

## Demos

| Demo             | Demonstrates                                                             | Pattern              |
| ---------------- | ------------------------------------------------------------------------ | -------------------- |
| `hello-world`    | Minimal command + dispatcher                                             | Inline manifest      |
| `deploy-tool`    | Typed flags (`--env`, `--verbose`, `--force`), colour, spinner           | Inline manifest      |
| `project-init`   | Flags + filesystem scaffolding from `ctx.cwd`                            | Inline manifest      |
| `api-client`     | Nested routes (`users/list`), env-driven config, **command discovery**   | File-based discovery |
| `todo-app`       | Multiple commands, positionals, persisted state, **command discovery**   | File-based discovery |
| `data-transform` | Reading stdin, format conversion (json/csv/table), **command discovery** | File-based discovery |
| `interactive-io` | **Logger**, **prompt/confirm/select**, spinners, colours, **stdin**      | File-based discovery |

## Running a demo

From the repo root (no install needed — demos import the framework by relative path):

```bash
bun run ./demos/hello-world/src/cli.ts hello Alice
# Hello, Alice!

bun run ./demos/deploy-tool/src/cli.ts deploy --env production --verbose
bun run ./demos/api-client/src/cli.ts users get 2
echo '{"name":"Alice"}' | bun run ./demos/data-transform/src/cli.ts format json
```

Build a standalone binary for any demo:

```bash
cd demos/hello-world
bun build ./src/cli.ts --compile --outfile dist/hello
./dist/hello hello Bob
```

## A few examples

```bash
# todo-app — state lives in ~/.watson-todos.json
bun run ./demos/todo-app/src/cli.ts add "Buy groceries"
bun run ./demos/todo-app/src/cli.ts list
bun run ./demos/todo-app/src/cli.ts complete 1

# deploy-tool — rollback is guarded
bun run ./demos/deploy-tool/src/cli.ts rollback --env staging          # exits 1: asks for --force
bun run ./demos/deploy-tool/src/cli.ts rollback --env staging --force  # proceeds

# api-client — configuration comes from the environment
API_URL=https://api.mysite.com bun run ./demos/api-client/src/cli.ts config

# interactive-io — logger, prompts, and colours
bun run ./demos/interactive-io/src/cli.ts list
echo '[{"id":"1","name":"Alice","email":"alice@example.com","role":"admin"}]' | bun run ./demos/interactive-io/src/cli.ts import
bun run ./demos/interactive-io/src/cli.ts view 1
```

## Validation

Every demo is exercised by a single black-box harness that spawns each CLI and
checks exit codes + key output. See [`tests/README.md`](../tests/README.md) for the
strategy. Run it with:

```bash
bun test tests/demos
```

## Manifests: Inline vs. Discovery-based

**Inline manifest** (`defineManifest`) — best for simple CLIs with a handful of commands:

```typescript
const manifest = defineManifest({ hello, goodbye })
```

**File-based discovery** (`discoverManifest`) — best for CLIs with many commands:

```typescript
const manifest = await discoverManifest(commandsDir)
```

With discovery, each command lives in its own `.ts` file under `commands/`, and the
router automatically discovers them. File structure mirrors routes: `commands/users/list.ts`
becomes the `users/list` command.

## Adding a Demo

Create `demos/<name>/src/cli.ts` and add an `EXPECTATIONS` entry in
`tests/demos/demos.test.ts` — the harness fails if a demo has no expectations.

For a discovery-based demo with multiple commands, create files in `commands/` (see
`api-client`, `data-transform`, `todo-app`, or `interactive-io` for examples).
