## Core Module: Runtime & Parser

The core module is CTI's command dispatch engine. It's where user input becomes command execution.

### Architecture

The core consists of three parts:

- **Runtime** (`runtime.ts`) — The main dispatcher: `defineManifest()` and `run()`
- **Router** (`router.ts`) — Internal helpers: route resolution and lookup building
- **Parser** (`parser.ts`) — Converts raw arguments into typed objects

The typical flow is: **argv → resolve route → parse flags → build context → invoke command**.

### Runtime

The runtime exports two functions that most code will use:

#### `defineManifest(routes: Record<string, CommandModule>): Manifest`

Builds a manifest from a route-to-module map. Routes are slash-delimited strings:

```typescript
const manifest = defineManifest({
  'hello': helloCommand,
  'users/list': usersListCommand,
  'users/get': usersGetCommand,
})
```

This is converted into a `Manifest` with entries that the router can consume. In a real project the manifest is code-generated by scanning your commands directory; demos build it inline so they stay in a single readable file.

#### `run(manifest: Manifest, config: Config, argv?: string[]): Promise<number>`

The main dispatcher. Resolves argv against the manifest, parses flags, builds the context, invokes the command, and returns an exit code:

```typescript
const exitCode = await run(manifest, config)
process.exit(exitCode)
```

Internally, `run()` performs these steps:

1. **Route resolution** — Find which command matches argv (longest-prefix match)
2. **Lazy loading** — Import the matched command module
3. **Flag parsing** — Parse and coerce argv into typed flags
4. **Context building** — Assemble the `Context` object
5. **Invocation** — Call `command.run(ctx)`
6. **Exit code handling** — Return the command's numeric result (or 0 if undefined)

If anything fails (command not found, parse error, exception during run), a terse error message is written to stderr and exit code 1 is returned.

### Router Helpers

The router module (`router.ts`) provides two functions that power the dispatcher:

#### `buildRouteLookup(manifest: Manifest): Map<string, ManifestEntry>`

Converts a manifest into a fast lookup map. Routes are joined with `/`:

```typescript
const lookup = new Map()
for (const entry of manifest.entries) {
  const routeKey = entry.route.join('/') // 'deploy/aws'
  lookup.set(routeKey, entry)
}
```

Why a map? **Performance.** Looking up a command is `O(1)` instead of scanning an array.

#### `resolveRoute(argv: string[], lookup: Map): { entry, remaining } | null`

Matches the longest possible route in argv against the lookup:

```typescript
function resolveRoute(args: string[], lookup: Map): Result | null {
  for (let i = args.length; i > 0; i--) {
    const routeKey = args.slice(0, i).join('/')
    const entry = lookup.get(routeKey)
    if (entry) {
      return { entry, remaining: args.slice(i) }
    }
  }
  return null
}
```

**Example:**

User runs: `my-cli deploy aws prod`

1. Check `deploy/aws/prod` in lookup → Not found
2. Check `deploy/aws` in lookup → **Found!** Return entry, remaining: `['prod']`

This allows nested commands (`my-cli deploy aws`, `my-cli deploy gcp`, etc.) and positional arguments.

#### Why Longest-Match?

```
my-cli deploy           → routes to 'deploy' command
my-cli deploy aws       → routes to 'deploy/aws' command (if it exists)
my-cli deploy aws prod  → routes to 'deploy/aws' command with positional 'prod'
```

This allows nested hierarchies and flexibility in how you organize commands. The longest match is tried first, so more-specific commands take precedence.

### Parser

The parser module (`parser.ts`) converts raw arguments into typed objects.

#### `parseAndCoerce(args: string[], flags: FlagSpec): ParseResult`

The main function. Takes argv and a flag specification, returns parsed and coerced values + positionals:

```typescript
// Input
argv = ['--environment=prod', '--verbose', 'src/', 'dist/']
flags = {
  environment: { type: 'string', default: 'staging' },
  verbose: { type: 'boolean', short: 'v' },
}

// Output
{
  values: { environment: 'prod', verbose: true },
  positionals: ['src/', 'dist/'],
}
```

Internally, it:

1. **Converts flags spec** to Node's `parseArgs` format
2. **Parses argv** using Node's built-in `parseArgs`
3. **Coerces values** to their declared types

#### Parsing Strategy

CTI uses Node.js's built-in `parseArgs` from `util`:

**Why Node's parseArgs?**

1. **No external dependencies** — Built into Node/Bun
2. **Battle-tested** — Used by Node's own CLI tools
3. **POSIX-compliant** — Handles GNU-style flags correctly
4. **Minimal API** — Does one thing well

#### Type Coercion

After parsing, values are coerced to their declared types:

```typescript
const coerced = {}
for (const [name, spec] of Object.entries(flags)) {
  const value = raw.values[name]
  if (value === undefined) continue
  coerced[name] = coerceValue(value, spec)
}
```

Coercion handles:

- **Strings** — Passed through as-is
- **Numbers** — Parsed with `Number()`, error if invalid
- **Booleans** — Already boolean from parseArgs

See [Utils: Coerce](utils.md) for details.

#### Flag Specifications

Commands declare flags using `FlagSpec`:

```typescript
interface FlagSpec {
  type: 'string' | 'boolean' | 'number'
  short?: string // Short flag (e.g., 'v' for -v)
  description?: string // Help text
  default?: Value // Default value if not provided
  required?: boolean // Must be provided
  multiple?: boolean // Can be specified multiple times
  choices?: string[] // Allowed values
  validate?: (v) => boolean | string // Custom validation
}
```

This is enough for most CLI tools. Complex validation can happen in the command handler.

### Design Decisions

#### Why Type Coercion?

Without coercion:

```typescript
// User types: my-cli delay --timeout=5000
ctx.flags.timeout === '5000' // String!
const wait = ctx.flags.timeout * 1000 // NaN
```

With coercion:

```typescript
// Parser knows timeout is 'number'
ctx.flags.timeout === 5000 // Number
const wait = ctx.flags.timeout * 1000 // 5000000 [Correct]
```

Coercion happens once, at parse time. Handlers don't need to check types.

#### Why Lazy Loading?

Commands are only imported when matched (via the importer in the manifest entry). A CLI with many commands shouldn't penalise quick invocations. If you run `my-cli help`, you don't want to load all 100 commands first.

Lazy loading means startup time is constant, regardless of how many commands are registered.

### Future Evolution

The core is stable but might evolve:

- **Validation hooks** — Run custom validation on flags before execution
- **Better error messages** — Suggest similar commands if not found (like Git)
- **Shell completion** — Generate bash/zsh/fish completion scripts

These would be additive, not breaking.
