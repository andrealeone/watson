# Manifest System

The manifest is a pre-generated or runtime-discovered registry of all available commands. It maps command routes to their implementations, enabling identical behavior in development and production (compiled binaries).

## What Is a Manifest?

A manifest is a data structure that lists all commands in your CLI:

```typescript
export interface Manifest {
  entries: ManifestEntry[]
}

export interface ManifestEntry {
  route: string[]              // e.g., ['db', 'migrate']
  sourcePath: string           // e.g., 'commands/db/migrate.ts'
  importer: () => Promise<...> // Lazy loader for the command module
  meta?: CommandMeta           // Optional metadata
}
```

Each entry maps a route to:

- The source file location
- A lazy importer function (for dynamic loading)
- Optional metadata (description, aliases, etc.)

## Development Mode

In development (running under `bun run`), the manifest is generated at runtime:

1. Watson walks the `commands` directory
2. Discovers all command files
3. Builds a manifest entry for each file
4. Routes incoming commands through the manifest

This allows instant feedback: add a file, and the command is available immediately.

## Production Mode

When compiled with `bun build --compile`, the manifest is pre-generated:

1. `watson build` walks the commands directory at build time
2. Generates a static manifest with hardcoded import paths
3. Embeds the manifest in the compiled binary
4. Routes work identically, but commands are already bundled

The compiled manifest looks like this:

```typescript
// Generated by watson build
export const manifest: Manifest = {
  entries: [
    {
      route: ['db', 'migrate'],
      sourcePath: 'commands/db/migrate.ts',
      importer: () => import('../commands/db/migrate.ts'),
    },
    {
      route: ['build'],
      sourcePath: 'commands/build.ts',
      importer: () => import('../commands/build.ts'),
    },
  ],
}
```

## Lazy Loading

Commands are loaded only when their route matches. If your CLI has 100 commands but the user runs one, only that command is imported. This keeps startup time fast.

## Single Source of Truth

The manifest is the single source of truth for available commands. Both the router and the help system read from it, so documentation always matches available commands.

## Identical Behavior

The genius of the manifest system is that the same code runs identically:

- Under `bun run` with a runtime-discovered manifest
- Inside a `bun build --compile` binary with a pre-generated manifest

The router and dispatcher never know which mode they're in.

## Command Metadata

Each manifest entry can carry metadata:

```typescript
export interface CommandMeta {
  description?: string
  aliases?: readonly string[]
  hidden?: boolean
  examples?: readonly string[]
}
```

This metadata is used for help generation and command discovery.

## Building the Manifest

The manifest is built by the discovery system:

## Never Hand-Written

The manifest is never hand-written or committed to version control. It's:

- Generated at runtime in development
- Generated at build time for production
- Excluded from `.gitignore`
- Transient and regenerable

This keeps your repository clean and eliminates manual synchronization.
